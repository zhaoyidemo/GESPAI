出现问题的地方

你的会话经常因 Claude 误解你的工作环境（本地与云）并采取错误的初始方法而脱轨，导致在你的 31 个会话中浪费了迭代周期。

错误的环境假设
Claude 反复假设你想要本地开发（克隆仓库、在本地安装依赖），但你实际上是在 GitHub/Railway 上进行云部署。你可以通过预先说明部署目标来防止这种情况——例如，"我通过 GitHub 部署到 Railway，不要在本地克隆或安装任何东西"——或将其添加到你的 CLAUDE.md 项目说明中。
  • Claude 在至少 4 个不同的场合试图在本地克隆仓库，而你想直接在 GitHub 上工作或部署到 Railway，迫使你每次都打断和重定向
  • Claude 建议 localhost URL 并尝试本地 npm/pip 依赖项安装，而你的工作流是基于云的，浪费时间并需要手动清理
需要多轮修复的不完整修复
Claude 经常在第一次尝试时提供部分修复——尤其是对于主题设置、构建错误和 API 配置等跨领域问题——迫使你进行 2-3 轮额外的相同请求。你可以要求 Claude 在提交之前系统地审计所有受影响的文件，或者在进行任何编辑之前请求所有需要更改的位置的检查清单。
  • 深色/浅色模式颜色修复要求你分别询问三次，因为 Claude 在第一次尝试时未能解决主题变体中的所有可读性问题
  • Judge0 API 配置错误（403/429 错误）和推送后的 TypeScript 构建失败需要多轮调试，你必须手动指出方法错误（例如，不必要地调用调试 API 端点）
误解的范围和能力
Claude 有时会误读你的要求——在你想要规划时开始构建代码，列出已安装的技能而不是所有可用的技能，或者尝试超出其能力的任务，如 3D 建模。对所需输出格式更明确（"给我一个计划，而不是代码"或"列出所有官方技能，而不仅仅是我安装的"）将减少这些失误。
  • 当你想讨论 PRD 和计划实施时，Claude 在你用实际需求文档打断之前直接开始构建通用项目，在另一个会话中误解了你想要 Scratch .sb3 文件而不是 Web 应用程序
  • Claude 对微信数据导出提出了太多连续的澄清问题，而不是预先呈现选项，导致你完全放弃了会话，没有形成任何具体计划

我需要你实现以下功能：[描述功能]。使用此循环自主工作：1) 在所有必要的文件中编写代码，2) 运行 `npm run build` 并修复任何 TypeScript 错误，3) 运行 `npm run lint` 并修复问题，4) 重复直到构建完全干净且零错误，5) 运行 `git diff` 并向我展示所有更改的摘要，6) 然后才要求我在提交之前审查。不要停下来问我构建错误的问题——自己修复它们。如果你三次遇到相同的错误，请解释是什么阻碍了你。
用于多文件重构的并行子代理
你跨 12+ 文件的命名约定更新和多文件主题/颜色修复都因不完整的第一次尝试而受到影响。Claude Code 可以生成并行子代理——每个文件或关注点一个——以执行具有跨文件一致性检查的协调重构。你成功的 GESP 测试修复会话已经非常有效地使用了子代理编排；这种模式应该成为任何涉及超过 3 个文件的更改的默认模式。
入门： 使用 Claude Code 的 TodoWrite 工具将大型重构分解为检查清单，然后指示 Claude 使用子代理进行并行执行。在更改后结合 Grep 验证所有文件的一致性。
粘贴到 Claude Code：
我需要在此项目的所有文件中重构 [描述更改]。遵循此过程：1) 使用 Grep 查找包含相关代码/模式的每个文件——不要遗漏任何文件，2) 创建一个 TodoWrite 检查清单，列出需要更改的每个文件，3) 使用并行子代理同时跨文件进行更改，4) 在所有更改完成后，运行验证：再次使用 Grep 查找任何剩余的旧模式，检查损坏的导入，并运行 `npm run build`，5) 向我展示完整的检查清单及每个文件的状态。目标是零遗漏文件——上次我们这样做时，你遗漏了前端显示名称和一个隐藏按钮。
用于 Bug 修复的测试驱动迭代
由于 14 个会话产生了有Bug的代码以及游戏逻辑中的竞态条件、过时坐标和深色/浅色模式回归等问题，你可以通过让 Claude 首先编写失败的测试，然后针对这些测试自主迭代代码更改，来显著提高第一次尝试的质量。你不需要在部署后可视化地发现双重刷新 bug，Claude 在你看到代码之前就在测试循环中捕获了它。
入门： 在你的 TypeScript 项目中设置一个最小的测试运行器（Vitest 或 Jest），并指示 Claude 在修复 bug 之前编写回归测试。Claude 可以使用 Bash 在循环中运行测试，直到所有测试通过。
粘贴到 Claude Code：
我有一个 bug：[描述 bug 和预期行为]。使用测试驱动开发修复它：1) 首先，编写一个（或多个）测试来重现确切的 bug——测试应该在当前代码下失败，2) 使用 `npm test` 运行测试以确认它因正确的原因失败，3) 现在修复实际代码，4) 再次运行测试——在你的修复上迭代直到测试通过，5) 添加一个边缘情况测试（例如，快速连续调用、主题切换、空状态），6) 运行完整的测试套件以检查回归，7) 只有在一切通过时才向我展示最终差异和测试结果。不要要求我手动验证——测试应该证明正确性。
